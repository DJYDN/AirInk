# 空中书写人机交互系统：项目需求与开发文档

版本：v0.1  
日期：2026-05-25  
定位：基于手部动作捕捉的轻量化、低延迟、可打包 Windows exe 的空中书写交互系统  
核心要求：所有开发、调试、测试、打包过程必须在隔离环境中完成，不能污染或破坏电脑原本系统环境。

---

## 1. 项目概述

### 1.1 项目背景

用户希望通过手部动作在空中完成“悬空书写”，电脑端能够实时同步显示书写轨迹，并支持画布、笔迹、背景、显示方式等自定义。最终系统应能够打包成 Windows exe 程序，便于部署、演示和后续产品化。

本项目不是单纯的手势识别，也不是普通摄像头画面显示，而是一个完整的人机交互系统：

```text
手部动作采集 → 手部关键点追踪 → 落笔/抬笔判断 → 轨迹滤波与映射 → 桌面画布实时渲染 → 保存/导出/打包
```

### 1.2 产品目标

1. 实现用户手部在空中移动时，电脑端实时显示轨迹。
2. 支持“落笔/抬笔”状态判断，避免手移动时一直乱画。
3. 保证系统轻量化、稳定可靠、低延迟、跟随性好。
4. 支持画笔、背景、画布、坐标映射、平滑程度等自定义。
5. 支持保存图片、保存轨迹数据，并最终打包为 Windows exe。
6. 所有测试必须隔离，不得影响系统原本环境、全局 Python 环境、系统注册表、系统摄像头设置、已有环境变量或用户文件。

### 1.3 推荐技术路线

推荐采用“视觉追踪为主、传感器辅助可扩展”的方案。

MVP 阶段采用：

```text
USB 摄像头 / 笔记本摄像头
+ OpenCV 视频采集
+ MediaPipe Hand Landmarker 手部关键点追踪
+ 捏合手势判断落笔/抬笔
+ One Euro Filter / Kalman Filter 轨迹滤波
+ PySide6 桌面端实时画布
+ PyInstaller 打包 exe
```

进阶阶段可加入：

```text
ESP32-S3 / ESP32-C3 指尖模块
+ IMU / PVC gel 传感器
+ USB 串口通信
+ 视觉-传感融合
+ 抗遮挡、低延迟预测、硬件落笔触发
```

## 3. 非功能需求

### 3.1 性能指标

| 指标 | MVP 目标 | 产品化目标 |
|---|---:|---:|
| 摄像头帧率 | ≥ 30 FPS | ≥ 60 FPS |
| 画布刷新率 | ≥ 30 FPS | ≥ 60 FPS |
| 端到端延迟 | ≤ 80 ms | ≤ 50 ms |
| 静止点抖动 | ≤ 5 px | ≤ 3 px |
| 落笔误触发率 | 可接受、可调 | 低误触发 |
| 连续运行时间 | ≥ 30 min | ≥ 2 h |
| exe 启动时间 | ≤ 10 s | ≤ 5 s |

说明：以上为产品目标，需要在目标电脑、目标摄像头、目标光照环境中实测，不能直接假定第三方库一定能达到。

### 3.2 稳定性要求

1. 摄像头断开后软件不能崩溃，应提示重新连接。
2. 手部短暂丢失时不应产生乱线。
3. 手部识别置信度低时应进入保护状态。
4. 配置文件损坏时应自动恢复默认配置。
5. 运行错误应写入项目本地日志，不应写入系统目录。

### 3.3 兼容性要求

| 项目 | 要求 |
|---|---|
| 操作系统 | Windows 10 / Windows 11 优先 |
| 摄像头 | 支持普通 USB 摄像头、笔记本摄像头 |
| 分辨率 | 640×480、1280×720、1920×1080 可选 |
| 打包形式 | 优先 one-folder，稳定后可尝试 one-file |
| 权限 | 不要求管理员权限 |

### 3.4 安全与隐私要求

1. 默认不保存摄像头原始画面。
2. 默认只保存用户主动导出的笔迹图片或轨迹数据。
3. 日志不得记录敏感图像内容。
4. 所有数据默认保存在项目本地用户数据目录，不写入系统敏感目录。
5. 若后续增加云同步，必须另行设计权限、加密和用户确认机制。

## 4. 环境隔离与测试安全要求

本项目所有测试必须遵守：**不能影响系统原本环境**。

### 4.1 总原则

1. 禁止在全局 Python 环境直接安装依赖。
2. 禁止使用 `pip install` 修改系统 Python 的 site-packages。
3. 禁止修改系统 PATH、注册表、系统摄像头配置、系统环境变量。
4. 禁止要求管理员权限运行测试。
5. 禁止测试脚本删除、覆盖用户非项目目录下的文件。
6. 所有测试输出必须写入项目目录下的 `tests/output/`、`logs/` 或临时沙盒目录。
7. 所有测试必须支持清理和回滚。
8. 硬件测试必须先支持 mock 模式，不能依赖真实硬件才能运行。

### 4.2 开发环境隔离方案

推荐使用项目本地虚拟环境：

```bash
python -m venv .venv
.venv\Scripts\activate
python -m pip install --upgrade pip
pip install -r requirements.txt
```

要求：

```text
项目根目录/
  .venv/                 # 本项目虚拟环境，不提交 Git
  requirements.txt       # 直接依赖
  requirements.lock.txt  # 锁定版本依赖
  src/                   # 源码
  tests/                 # 测试
  logs/                  # 本地日志
  data/                  # 本地数据
  dist/                  # 打包输出
```

`.gitignore` 必须包含：

```gitignore
.venv/
__pycache__/
*.pyc
logs/
data/runtime/
tests/output/
dist/
build/
*.spec
.env
```

### 4.3 依赖管理要求

1. 使用 `requirements.txt` 管理直接依赖。
2. 使用 `requirements.lock.txt` 锁定可复现版本。
3. 新增依赖必须说明用途，不能随意引入大型库。
4. 测试依赖单独放入 `requirements-dev.txt`。
5. 打包依赖单独记录，避免开发依赖进入最终包。

示例：

```text
requirements.txt
  opencv-python
  mediapipe
  PySide6
  numpy
  pyserial
  pydantic

requirements-dev.txt
  pytest
  pytest-qt
  pytest-cov
  black
  ruff
  mypy
```

### 4.4 测试沙盒目录

所有测试只能写入以下目录：

```text
项目根目录/tests/output/
项目根目录/tests/tmp/
项目根目录/logs/test/
系统临时目录下的 airwrite_test_xxx/
```

禁止写入：

```text
C:\Windows\
C:\Program Files\
C:\Users\用户\Desktop\
C:\Users\用户\Documents\
系统注册表
全局 Python site-packages
全局环境变量
```

除非用户在 UI 中主动选择导出路径，否则程序不得自行写入桌面、文档、下载等用户目录。

### 4.5 环境变量约束

运行模式通过本进程环境变量控制，不写入系统环境变量：

```bash
set AIRWRITE_ENV=test
set AIRWRITE_CONFIG_DIR=./tests/output/config
set AIRWRITE_DATA_DIR=./tests/output/data
set AIRWRITE_LOG_DIR=./tests/output/logs
```

程序启动时根据环境变量决定配置目录：

| 模式 | 配置路径 | 说明 |
|---|---|---|
| dev | `./data/dev/` | 本地开发 |
| test | `./tests/output/` | 自动测试 |
| prod | 用户 AppData 或程序目录 | 正式运行 |

### 4.6 摄像头测试隔离

摄像头测试分为三类：

| 类型 | 输入 | 是否影响系统 |
|---|---|---|
| 单元测试 | 预录制视频 / 图片序列 | 不影响 |
| 集成测试 | 虚拟 CameraSource | 不影响 |
| 手动测试 | 真实摄像头 | 仅运行时读取，不修改系统配置 |

要求：

1. 自动化测试默认不打开真实摄像头。
2. 使用 `MockCameraSource` 或本地测试视频代替摄像头。
3. 真实摄像头测试必须人工触发。
4. 程序退出时释放摄像头资源。
5. 不修改摄像头驱动、系统默认摄像头、系统隐私设置。

### 4.7 串口/硬件测试隔离

1. 自动化测试使用 `MockSerialDevice`。
2. 不自动扫描并写入未知串口设备。
3. 真实硬件测试需要用户手动选择串口。
4. 串口通信默认只读，发送控制命令必须用户确认。
5. 固件烧录不纳入桌面端自动测试流程。

### 4.8 打包测试隔离

1. 打包必须在 `.venv` 中执行。
2. 打包输出仅写入 `build/` 和 `dist/`。
3. 测试 exe 时使用测试配置目录，不读取或覆盖正式配置。
4. 打包测试不创建系统服务、不写注册表、不添加开机自启。
5. 卸载测试只删除项目自己的安装目录和配置目录，不扫描删除其他文件。

## 6. 技术选型

### 6.1 核心技术

| 类别 | 推荐 | 原因 |
|---|---|---|
| 开发语言 | Python | 快速开发、生态成熟、便于算法验证 |
| 摄像头读取 | OpenCV | 摄像头读取、帧处理方便 |
| 手部追踪 | MediaPipe Hand Landmarker | 可输出手部关键点、世界坐标、左右手信息 |
| 桌面 UI | PySide6 / Qt for Python | 适合开发 Windows 桌面端，可做复杂画布 |
| 绘制 | QPainter / QGraphicsView | 适合实时轨迹绘制和自定义显示 |
| 数值处理 | NumPy | 坐标计算、滤波、轨迹处理 |
| 串口通信 | pyserial | 后续接入 ESP32 硬件 |
| 打包 | PyInstaller | 可将 Python 程序与依赖打包为 exe |
| 测试 | pytest | 单元测试、集成测试、自动化验证 |

### 6.2 为什么不建议纯 IMU 方案作为主方案

纯 IMU 需要通过加速度二次积分获得位移，悬空书写时会产生明显漂移，长时间书写轨迹难以稳定闭合。IMU 更适合作为短时预测、姿态判断、落笔状态辅助，而不适合作为唯一位置来源。

### 6.3 为什么采用视觉作为 MVP 主方案

1. 不需要制作硬件，开发成本低。
2. 能直接得到图像平面中的手部关键点。
3. 适合快速验证交互流程。
4. 便于后续加入传感器融合。

## 7. 关键算法设计

### 7.1 手部关键点

重点使用以下关键点：

| 关键点 | 用途 |
|---|---|
| 食指指尖 | 作为笔尖位置 |
| 拇指指尖 | 与食指距离判断捏合 |
| 食指根部 | 判断食指方向、辅助稳定性 |
| 手腕 | 坐标归一化、手部尺度估计 |
| 掌心区域 | 判断手部整体稳定性 |

### 7.2 落笔/抬笔状态机

```text
NO_HAND：未检测到手
  ↓ 检测到手
HOVER：悬停，光标跟随但不绘制
  ↓ 捏合持续 N 帧
DRAWING：落笔绘制
  ↓ 捏合释放持续 M 帧
HOVER：回到悬停
  ↓ 手丢失
LOST：短时丢失，冻结或预测
  ↓ 超过阈值
NO_HAND：停止绘制
```

### 7.3 捏合判断

不能使用固定像素距离，因为手离摄像头远近会改变尺度。应使用归一化距离：

```text
pinch_distance = distance(index_tip, thumb_tip)
hand_scale = distance(wrist, middle_finger_mcp)
normalized_pinch = pinch_distance / hand_scale
```

判断逻辑：

```text
normalized_pinch < down_threshold → 可能落笔
normalized_pinch > up_threshold   → 可能抬笔
```

其中 `down_threshold` 小于 `up_threshold`，形成迟滞，避免在临界值附近频繁抖动。

### 7.4 坐标映射

MVP 可使用简单线性映射：

```text
canvas_x = image_x / image_width  * canvas_width
canvas_y = image_y / image_height * canvas_height
```

产品化阶段使用四点透视标定：

```text
摄像头书写区域四点 → 画布四点
通过透视变换建立映射关系
```

### 7.5 轨迹滤波

推荐实现两种滤波器：

1. One Euro Filter：适合交互系统，低速时抑制抖动，高速时降低滞后。
2. Kalman Filter：适合加入速度估计和短时预测。

UI 中提供“平滑强度”滑条，不让用户直接理解复杂参数。

### 7.6 丢帧处理

| 情况 | 处理方式 |
|---|---|
| 丢失 1–3 帧 | 使用上一帧位置或线性预测 |
| 丢失 3–10 帧 | 冻结光标，不继续绘制 |
| 丢失超过阈值 | 自动抬笔，避免产生乱线 |
| 重新检测到手 | 回到 HOVER，等待重新落笔 |

## 11. 开发计划

### 11.1 阶段 0：环境与工程骨架

目标：建立不污染系统的开发环境。

任务：

1. 创建项目目录。
2. 创建 `.venv` 虚拟环境。
3. 编写 `requirements.txt` 和 `requirements-dev.txt`。
4. 建立目录结构。
5. 配置日志路径、配置路径、测试输出路径。
6. 编写 mock 摄像头和 mock 串口接口。

验收：

1. 不使用全局 Python 安装依赖。
2. 删除 `.venv` 后系统环境不受影响。
3. 自动测试输出只出现在 `tests/output/`。

### 11.2 阶段 1：视觉追踪 MVP

目标：实现手部关键点检测和食指轨迹显示。

任务：

1. 摄像头读取。
2. MediaPipe 手部关键点检测。
3. 食指指尖坐标提取。
4. 坐标映射到画布。
5. PySide6 主窗口和画布。
6. 显示实时轨迹。

验收：

1. 正常光照下食指移动，画布光标能够跟随。
2. FPS ≥ 30。
3. 程序关闭后摄像头释放。

### 11.3 阶段 2：落笔与轨迹优化

目标：实现可用的悬空书写。

任务：

1. 捏合落笔/抬笔判断。
2. 状态机实现。
3. One Euro Filter 或 Kalman Filter。
4. 曲线平滑。
5. 丢帧保护。
6. 清屏、撤销、保存 PNG。

验收：

1. 捏合时绘制，松开时停止。
2. 不捏合时移动手不会乱画。
3. 轨迹抖动明显降低。
4. 手部短暂丢失不产生乱线。

### 11.4 阶段 3：产品化界面

目标：形成可演示软件。

任务：

1. 参数面板。
2. 摄像头选择。
3. 画笔颜色、粗细、背景设置。
4. 四点标定。
5. 配置保存和恢复。
6. FPS、延迟、置信度显示。
7. 新手引导。

验收：

1. 用户无需改代码即可调整参数。
2. 重启软件后配置仍然有效。
3. UI 不阻塞实时追踪。

### 11.5 阶段 4：打包与发布

目标：生成 Windows exe。

任务：

1. 编写 PyInstaller 打包脚本。
2. 处理模型文件、图标、配置默认值。
3. one-folder 打包。
4. 干净环境测试 exe。

## 13. 打包与发布方案

### 13.1 打包策略

优先使用 one-folder：

```bash
pyinstaller --noconfirm --onedir --windowed src/airwrite/main.py
```

原因：

1. 启动速度通常更稳定。
2. 资源文件和模型文件更容易排查。
3. 出问题时便于定位缺失依赖。
4. 对早期项目更适合。

后期再考虑 one-file：

```bash
pyinstaller --noconfirm --onefile --windowed src/airwrite/main.py
```

### 13.2 资源管理

需要打包的资源：

```text
resources/models/
resources/icons/
resources/themes/
默认配置文件
```

要求：

1. 程序内统一使用 `resource_path()` 获取资源路径。
2. 不使用绝对路径。
3. 不依赖开发机路径。
4. 打包后必须验证模型文件能被正确加载。

AirWrite_v0.1.0/

## 14. 风险分析与应对

| 风险 | 表现 | 应对 |
|---|---|---|
| 光照不足 | 手部识别不稳定 | 新手引导提示补光，显示置信度 |
| 背景复杂 | 误识别 | 提供 ROI 限制、建议纯色背景 |
| 手指抖动 | 笔迹毛刺 | 滤波、曲线平滑、参数可调 |
| 滤波过强 | 跟随滞后 | 平滑强度可调，快速移动降低滤波 |
| 捏合误判 | 乱落笔 | 归一化阈值 + 迟滞 + 连续帧确认 |
| 手部遮挡 | 轨迹中断 | 短时冻结/预测，后续接传感器 |
| UI 卡顿 | 延迟高 | 采集/处理/UI 分线程，避免主线程阻塞 |
| 打包失败 | exe 无法启动 | one-folder 优先，资源路径统一管理 |
| 测试污染系统 | 破坏原环境 | 全部测试沙盒化，禁止全局安装和系统写入 |
